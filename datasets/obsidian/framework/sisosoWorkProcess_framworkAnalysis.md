# 로컬 스테이 웹 콘텐츠 설계 프로세스 v1.0.1

## 0. 전제

- 목표:
    
    - “시설을 파는 웹사이트”가 아니라, **로컬 전체를 파는 웹사이트**의 콘텐츠 구조를 설계한다.
    - 기존 아키텍처(효우당/무카유 CSV 스키마)를 유지하면서,  
        로컬 프로젝트에 맞게 카피/구조를 재작성한다.
- 입력물:
    
    - 기존 CSV들: `items.csv`, `subitems.csv`, `tag0.csv`, `tags.csv`, `stories.csv`
    - PROTOCOL.json (Airtable/Claud 파이프라인 스펙)
    - 사용자의 컨셉 디렉션:
        - “로컬의 로컬을 판다”
        - “컨셉을 설명하지 말고, 컨셉에 충실하게 대상(풍경, 식재료, 사람)을 말하기”
        - “방/식사/로비/체험/지도”를 매거진처럼 내러티브로 엮기

## 1. 기존 구조 이해 단계

### 1-1. CSV 스키마 파악

- `tag0.csv`
    
    - 카테고리/섹션 정의
    - 예: 객실, 어메니티, 오퍼, 시설, globals 등
    - `items` 컬럼에 해당 섹션에 속한 item들의 `_record_id` 배열
- `items.csv`
    
    - 모든 컨텐츠 블록이 한 테이블에 모여 있음
    - 필드: `_record_id`, name, subtitle, notes, notes2, price, caption, db, tag0, tags, subitems, layout, order, ctaText, img, gallery1~10
    - `db`/`layout`으로 타입/표현 방식을 바꿈
- `subitems.csv`
    
    - 어떤 item에 붙는 “조각 정보”
    - 예: 포함 내용, 신청 오픈, 스테이 가능 기간, 신용카드, 온천 안내 등
- `tags.csv`
    
    - 태그 정의
    - Food / Culture / spring / summer / offer 등
    - `items` 컬럼에 태그가 붙는 item들
- `stories.csv`
    
    - 스토리/에세이 컨텐츠 (이번 프로젝트에서는 크게 안 활용)

### 1-2. PROTOCOL.json 이해

- 타입 정의:
    
    - TXT → singleLineText
    - LNG → multilineText
    - SEL → singleSelect
    - LNK → multipleRecordLinks (target 필요)
- schema_spec 룰:
    
    - 각 table은 CSV 파일과 1:1 연결
    - primary_key로 링크 매칭
    - LNK 필드는 `target`으로 다른 table을 가리킴

**결론:**  
기존 아키텍처(tag0/items/subitems/tags + PROTOCOL 타입체계)를 그대로 두고,  
내용과 구조(카테고리/아이템/관계)만 로컬 프로젝트에 맞게 갈아끼우기로 결정.

## 2. 개념 설계 단계 — “어떤 웹을 만들 것인가”

### 2-1. 섹션 정의

사용자 디렉션을 바탕으로, 페이지를 다섯 축으로 나눔:

1. 객실 (Rooms)
2. 식사 (Food)
3. 로비 (Lobby / Shop)
4. 체험 (Experiences)
5. 이웃들 / 네이버후드 (Neighborhood)

추가로, 이 위에 항상 깔려야 하는 전역 정보 레이어:

6. 글로벌 인포 (Globals)
    - 프로젝트 소개 / 이용 안내 / 문의 / 예약·취소 정책

### 2-2. 카피 방향성 합의

- 컨셉을 설명하지 않는다.  
    → “우리는 로컬의 로컬을 팝니다”를 직접 말하지 않고,  
    방/식사/로비/체험/이웃들 설명 자체에서 그 느낌이 나게 만든다.
    
- 방:
    
    - “바다/파도/수평선”은 카테고리 오버뷰에서 한 번만 말하고,
    - 개별 방(아이템)은 각 층/각도의 차이만 이야기한다.
    - 시설 스펙은 최소한 한 줄로만.
- 식사:
    
    - “어떤 코스인지” 대신, “오늘 바다·산에서 들어온 재료”와  
        봄/여름/가을/겨울에 따라 어떻게 상이 달라지는지 **메뉴판 형식**으로 구체적으로.
- 로비:
    
    - 시설 설명 X
    - “로컬 파트너들이 만든 물건” 자체만 이야기.
    - 아이템은 실제 온라인 샵으로도 연결될 수 있도록  
        (타이틀/내러티브/디테일/가격/CTA) 구조를 가진다.
- 체험:
    
    - 새벽 바다 → 아침 시장 → 저녁 모닥불로 이어지는 “하루” 내러티브.
    - 각 체험은 내러티브 + 디테일(일정/인원/포함사항/가격/예약).
- 이웃들:
    
    - 지도에 점 찍는 게 목표가 아니라,
    - “로비/체험에서 봤던 파트너들이 실제 어디서 무엇을 하는지”를 이어주는 레이어.
    - 각 이웃은 설명 + 정보 리스트 + 사이트 URL을 가진다.

## 3. 카피 초안 작성 — 섹션별 내러티브 쓰기

이 단계에서는 아직 CSV 생각은 접어두고,  
그냥 “페이지에 들어갈 문장”으로 초안을 쭉 작성.

### 3-1. Rooms

- 카테고리 오버뷰:
    
    - 이곳의 방에서는, 어디에 머물든 창을 통해 삼척 앞바다의 파도와 수평선을 한눈에 보게 됩니다.
- 방1: 바다 숨소리가 가까운 방
    
    - 파도가 바로 아래에서 부서지는 소리가 잘 들리는 층
    - 기본 정보: 기준 2인 / 더블 / 욕실 1개
- 방2: 수평선이 넓게 열린 방
    
    - 눈높이에서 수평선이 곧게 이어지는 높은 층
    - 기본 정보: 기준 2인 / 트윈 / 욕실 1개

### 3-2. Food

- 오버뷰:
    
    - “정해진 코스”가 아니라, 오늘 바다/산에서 들어온 재료가 먼저인 식사.
- 카테고리 컨텐츠: “오늘의 상차림” (칠판에 적힌 재료 목록 + 그날 상차림)
    
- 봄/여름/가을/겨울 메뉴를
    
    - 각 섹션 제목 + 한두 줄 설명 + **구체적인 메뉴 리스트**로 기술
    - 예: 봄 — 봄 해초 모둠, 들나물 된장국, 진달래 화전, 봄 채소와 생선구이, 쑥밥

### 3-3. Lobby

- 오버뷰:
    
    - 로비에는 로컬 파트너들이 매일 손으로 만드는 결과물(김, 약초 티, 공예품 등)을 소개.
- 카테고리 컨텐츠:
    
    - “이 동네의 손끝에서 나온 것들” — 물건과 사람을 중심으로 설명.
- 아이템 3개:
    
    - 동해 갓마른 김
    - 산자락 약초 블렌드 티
    - 파도결 나무 트레이

각각:

- 내러티브: 이게 어떤 맥락/손에서 나왔는지
- 디테일 리스트: 구성/원산지/재료/사용법
- 가격(예시)
- CTA 텍스트

### 3-4. Experiences

- 오버뷰:
    
    - 로컬 파트너들의 일상(바다/시장/공방/모닥불)을 따라가는 경험.
- 카테고리 컨텐츠:
    
    - “하루를 통째로 맡겨보는 방법” — 새벽·아침·저녁 프로그램을 이어서 체험하면 로컬의 하루를 거의 그대로 따라가게 됨.
- 체험 3개:
    
    - 새벽 바다 산책
    - 아침 수산시장 동행
    - 저녁 바닷가 모닥불

각각:

- 내러티브 (감각 중심)
- 디테일 리스트:
    - 일정(시간/모임 장소/이동수단)
    - 인원 제한
    - 포함 사항
    - 준비물
    - 가격(예시)
    - CTA 텍스트

### 3-5. Neighborhood

- 오버뷰:
    
    - 로비/체험에서 만난 이름들(어부, 시장, 공방)을  
        실제 지도와 정보로 연결하는 섹션.
- 카테고리 컨텐츠:
    
    - “바다와 마을 사이의 작은 점들” — 포구/시장/공방/책방이 점으로 찍힌 지도 이미지 설명.
- 이웃 3개:
    
    - ○○항 작은 포구
    - △△ 수산시장
    - □□ 공방 & 동네 책방

각각:

- 내러티브 (어떤 역할을 하는지)
- 디테일 리스트 (위치, 하는 일/제공 서비스, 추천 시간대, 비고)
- 외부 웹사이트 링크를 subitems로 연결

## 4. globalInfo 설계 — 전역 인포 분리

### 4-1. 글로벌 인포 항목 정의

글로벌 레벨에서 항상 필요한 정보:

1. `global_about` — 프로젝트 소개
2. `global_guide` — 이용 안내
3. `global_contact` — 문의/연락처
4. `global_policy` — 예약/취소/환불 정책

### 4-2. 카피 작성

- About:
    - “개별 호텔이 아니라, 이 동네 전체를 소개하는 프로젝트”
- Guide:
    - 체크인/체크아웃, 조용한 시간, 흡연/반려동물, 인터넷, 주차
- Contact:
    - 전화/이메일/인스타그램
- Policy:
    - 7일 전 전액 환불 / 3일 전까지 50% / 이후 환불 불가 (예시)

### 4-3. 섹션 연결

- `tag0.globals_page`를 만들고
    - `items = ["global_about","global_guide","global_contact","global_policy"]`
- `items.global_*`는 `tag0 = ["globals_page"]`로 설정
- 필요하면 `tags`에 `tag_globalInfo` 태그를 붙여서 전역 모달/푸터 등에 사용

## 7. 체크리스트 — 다음 번에 이 프레임워크 쓸 때

### 7-1. 준비

- 기존 CSV 스키마(or 기준 스키마) 확보
- PROTOCOL.json 확인
- 프로젝트 컨셉/섹션 정의 (e.g. Rooms/Food/Lobby/Experiences/Neighborhood)

### 7-2. 개념 정리

- 섹션별로:
    - 공통 성질(카테고리에서 한 번만 말할 것)
    - 아이템별로 다르게 말할 수 있는 포인트
- globalInfo로 빼야 할 것:
    - About / Guide / Contact / Policy

### 7-3. 카피 작성

- CSV 생각 없이, 섹션별 내러티브 & 디테일 문장부터 작성
- “컨셉 설명 문장”은 최대한 제거하고,  
    컨셉에 충실한 설명만 남기는 작업 한 번 더 하기

### 7-4. CSV 매핑

- tag0: 섹션 정의 + items 배열
- items: 카피/디테일/CTA/타입(db)/레이아웃(layout)
- subitems: 필요하다면 링크/포함내용/기간 등 조각 정보
- tags: 시즌/카테고리/글로벌 인포 태그

### 7-5. 스키마 정의

- SCHEMA_x.y.z.json에:
    - csv 파일명
    - primary_key
    - fields 타입 + 링크 target
