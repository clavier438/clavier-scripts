## 3. 도요타식 칸반 — 핵심 원칙 요약

**1) 흐름(Flow) 관리가 핵심**

- "뭐가 남았지?"가 아니라 \*\*"어디에서 막히고 있지?"\*\*에 집중
- 보드는 To-Do 리스트가 아니라 병목 탐지 도구

**2) Pull(당겨오기) 시스템**

- 위에서 억지로 밀어 넣는 Push가 아니라
- 내 칼럼에 빈자리가 생겼을 때 이전 단계에서 다음 카드를 당겨오는 방식

**3) WIP Limit — 동시 진행 개수 제한**

- 각 칼럼에 "최대 카드 수"를 명시적으로 정한다
- 넘으면 새 카드 못 넣음 → 병목 원인 해결부터

**4) 카드 = 작업 지시서 + 문제 탐지 장치**

- 최소 정보: 호텔 이름 / 담당자 / 현재 단계 / 다음 액션 1개 / 금액 / 시작일·런칭일

**5) 정기 보드 리뷰 = 카이젠(지속 개선)**

- 매일/매주 보드 앞에서 병목 찾고, WIP·칼럼·프로세스를 조정하는 루프

## 7. 오전 4시간 루틴 (칸반 기준)

**1단계: 보드 체크 (10\~15분)**

- Trello(또는 Jira) 보드 열기
- 각 칼럼 카드 수 확인 → WIP 넘는 곳 있는지
- 3일 이상 같은 칼럼에 멈춘 카드 있는지

**2단계: 병목 해결 (1\~1.5시간)**

- 오래 멈춘 카드부터 처리
  - 답 없는 메일 리마인드
  - 자료 요청 재촉
  - 견적서 마무리
- WIP 초과 칼럼 → 원인 파악 후 해결

**3단계: Pull — 새 카드 당겨오기 (1.5\~2시간)**

- 빈자리 생긴 칼럼에서 이전 단계 카드를 당겨와서 작업 시작
- 예: 제작 칼럼 빈자리 → 계약 완료된 호텔 카드 당겨서 사이트 세팅 시작

**4단계: 다음날 준비 (10\~15분)**

- 오늘 이동시킨 카드 정리
- 내일의 Next Action 1줄씩 업데이트
- 보드 스냅샷으로 "오늘 뭘 했는지" 가시화

## 9. 빌드 인 퍼블릭 (Build in Public) 전략

### 9-1. 왜 하는가

| 효과 | 설명 |
| --- | --- |
| **자기 압박** | 공개했으니 멈출 수 없다. "이번 주 진전 0"을 쓰고 싶지 않은 마음이 최고의 동력 |
| **주목도** | "오전 4시간만 일하면서 월 500 만드는 과정"은 사람들이 진짜 궁금해하는 스토리 |
| **인바운드 리드** | 호텔 사장님이 콘텐츠를 보고 "나도 웹사이트 필요한데" 하고 먼저 연락 옴 |
| **신뢰 구축** | 과정을 투명하게 공개하면 "이 사람은 진짜 하는 사람"이라는 인식이 생김 |
| **네트워크** | 비슷한 1인 사업자, 프리랜서, 노코드 커뮤니티에서 연결이 생김 |

**메인 채널 1개 + 보조 1개** 로 시작 (너무 많으면 콘텐츠 제작 자체가 병목이 됨)

### 9-6. 빌드 인 퍼블릭 + 칸반 연동

칸반 보드를 운영하면 빌드 인 퍼블릭 콘텐츠가 **자동으로** 만들어짐:

| 칸반 이벤트 | → 콘텐츠 |
| --- | --- |
| 카드가 "컨택 발송"으로 이동 | "오늘 N곳에 제안 메일 발송" |
| 카드가 "상담 중"으로 이동 | "첫 미팅 성사! N번째 리드" |
| 카드가 "계약 진행"으로 이동 | "계약 N호 진행 중. MRR +Xk" |
| WIP 초과 발생 | "병목 발견. 상담 2건이 동시에 막힘. 원인 분석 중" |
| 카드가 "런칭 완료"로 이동 | "N번째 호텔 사이트 런칭! 스크린샷 공유" |

> **결론:** 칸반을 제대로 쓰면 콘텐츠를 따로 "만들" 필요가 없다. 보드에서 일어나는 일을 그대로 공유하면 됨.
