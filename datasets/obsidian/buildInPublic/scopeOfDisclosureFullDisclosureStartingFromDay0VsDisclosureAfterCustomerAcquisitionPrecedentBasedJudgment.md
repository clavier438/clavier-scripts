#BuildInPublic 

> 부티크 호텔 웹사이트 1인 사업 | 2026.04.08\
> "에이전시 포트폴리오처럼 처음부터 다 밝힐까, 아니면 고객 좀 잡고 나서 열까?"

## 1. 이미 해본 사람들이 뭐라고 하는가

### Arvid Kahl (FeedbackPanda 창업·엑싯, 빌드인퍼블릭 방법론 저자)

**요약: "처음부터 열어라. 단, AI 시대에는 공유할 내용의 종류를 바꿔라."**

Arvid는 빌드인퍼블릭을 "잠재 고객, 기존 고객, 동료 파운더 앞에서 비즈니스를 만드는 여정을 공유하는 것"이라고 정의한다. 핵심은 실패와 실험이 잘못되는 것을 인정하는 게 진짜 신뢰 기반의 장기적 관계를 만든다는 것.

**단, 최근 경고:** Arvid는 최근 팟캐스트에서 "빌드인퍼블릭이 FeedbackPanda를 파는 데 도움을 줬지만, 같은 급진적 투명성이 이제는 비즈니스를 하루아침에 망칠 수 있다"고 말했다. AI 코딩 도구가 등장하면서, 공개적으로 공유한 아키텍처나 기능 상세를 경쟁자가 며칠 만에 복제할 수 있게 됐기 때문. **하지만 공유 자체를 멈추라는 게 아니라, "무엇을" 공유하느냐를 바꾸라는 뜻.**

→ **너의 상황에 적용:** 서비스업(웹사이트 제작)은 SaaS와 다르다. 구조를 공개해도 "똑같이 실행"할 수 있는 사람이 거의 없다. 리스크가 SaaS보다 훨씬 낮다.

### Pieter Levels (@levelsio, Nomad List·Photo AI, 월 $300K+)

**요약: "처음부터 열어라. 스토리 자체가 마케팅이다."**

Levels는 자신의 인생과 프로덕트를 만드는 과정의 우여곡절을 "주된 마케팅 수단"으로 썼다고 직접 밝혔다. "완벽한 스타트업 파운더처럼 행동하는 대신, 최대한 투명하고 취약하게 공유하려 했다. 사람들은 불완전함과 실수를 공유하면 함께 여정을 가고 싶어한다."

그의 접근법 핵심은: 최소한의 것을 만들어서 즉시 공개하고, 사람들의 반응을 보면서 개선하는 것. "프로덕트가 스토리가 된다. 별도 마케팅 없이."

→ **너의 상황에 적용:** "오전 4시간, 월 500만 원" 자체가 사람들이 궁금해하는 스토리. Levels의 "12 startups in 12 months"처럼, 구조 자체가 콘텐츠가 된다.

### Founder Reality (실전 파운더 블로그)

**요약: "진짜 빌드인퍼블릭은 좋은 것만 올리는 게 아니다. 타이밍이 중요하다."**

이 블로거는 "대부분의 빌드인퍼블릭은 좋은 부분만 보여주는 스타트업 극장"이라고 비판하면서, 진짜 투명성은 Buffer처럼 좋은 것과 나쁜 것 모두 공개하는 것이라고 말한다. 실전 조언으로 "큰 사건을 실시간으로 트윗하지 마라. 처리할 시간을 줘라"고 했다.

→ **너의 상황에 적용:** 계약 실패나 거절 같은 걸 바로 올리기보다, 며칠 후에 "배운 점" 프레임으로 올리는 게 더 효과적.

### Mercury (핀테크 스타트업 뱅킹)

**요약: "공개인가 비공개인가는 이분법이 아니라 스펙트럼이다."**

Mercury의 가이드는 핵심을 잘 짚는다: "목표는 최대한의 투명성이 아니라, 지속 가능한 명확성(sustainable clarity)이다." 초기에는 실험·배움·판단 근거를 공유하고, 매출 숫자 같은 건 나중에 공유해도 된다.

→ **너의 상황에 적용:** 전부 공개 vs 전부 비공개가 아니라, "지금 어떤 종류의 정보를 열 것인가"를 단계적으로 결정.

## 2. 두 가지 전략 비교 — 득과 실

### 전략 A: Day 0부터 에이전시 포트폴리오처럼 전부 공개

**"부티크 호텔 30곳 리서치 → 이런 문제 발견 → 이런 구조로 해결 → 이런 결과"**

| 얻는 것 | 왜 |
| --- | --- |
| **인바운드 리드** | 호텔 사장님이 콘텐츠를 보고 "나도 해달라"고 먼저 연락. 에이전시의 케이스 스터디가 가장 강력한 영업 도구인 이유가 이것. |
| **전문가 포지셔닝** | "부티크 호텔 = 이 사람"으로 니치 연결. 빌드인퍼블릭에서 가장 공개적인 사람이 그 니치의 대표가 된다. |
| **자기 압박** | 공개했으니 멈출 수 없다. 혼자 하는 사업에서 이게 최고의 동력. |
| **피드백 사이클** | 구조를 공개하면 "이 부분은 이렇게 하는 게 낫지 않냐"는 피드백이 온다. |
| **신뢰** | 과정을 투명하게 보여주면 가격 협상에서 유리. "이 사람은 진짜 이 분야를 연구한 사람"이라는 인식. |
| **네트워크** | 비슷한 1인 사업자, 노코드 커뮤니티와 연결. 소개·레퍼럴 발생. |

| 잃을 수 있는 것 | 왜 |
| --- | --- |
| **가격 앵커링** | 구축비를 공개하면, 그보다 싸게 하겠다는 경쟁자가 나올 수 있음 |
| **실패의 공개** | 계약 거절, 매출 0 같은 걸 공개하면 "이 사람 실력 없나?" 오해 가능 |
| **콘텐츠 부담** | 아직 보여줄 결과물이 없는 상태에서 "뭘 보여주지?"하는 고민 |
| **경쟁자 교육** | 니치 정의·문제 분석·구조를 공개하면 경쟁자가 베낄 수 있음 |

### 전략 B: 고객 2\~3곳 확보 후 Before/After와 함께 공개

**"조용히 첫 계약 → 런칭 → 결과 확보 → 그때 케이스 스터디로 공개"**

| 얻는 것 | 왜 |
| --- | --- |
| **증거 기반 공개** | Before/After 스크린샷, 실제 수치(예약 증가, 직예약 비율 등)가 있으면 설득력 10배 |
| **피벗 자유** | 공개 전에 가격·구조·타겟을 조용히 바꿀 수 있음. "왜 바꿨어?"라는 질문 안 받음 |
| **완성도 높은 첫 인상** | 첫 공개 시점에 포트폴리오가 이미 있으므로, "이 사람은 이미 하고 있는 사람"이라는 인식 |
| **고객 프라이버시** | 초기 고객과 조건 협상 시 공개 압박 없음 |

| 잃을 수 있는 것 | 왜 |
| --- | --- |
| **0→1 기간의 고독** | 혼자 하는 사업에서 외부 압박 없이 쉽게 미루게 됨 |
| **오디언스 빌딩 지연** | 3\~6개월 조용히 일하면, 그 기간 동안 팔로워/오디언스 0. 나중에 공개해도 아무도 안 봄 |
| **인바운드 리드 0** | 공개 전까지 모든 리드를 아웃바운드로만 잡아야 함. 더 힘듦 |
| **기회비용** | "여정의 시작"은 한 번뿐. 나중에 공개하면 "Day 0의 설렘"을 재현할 수 없음 |

## 3. 네 상황에 맞는 핵심 판단 기준

### "경쟁자가 베낄 수 있는가?" — 이게 핵심 질문

Arvid Kahl의 경고는 **SaaS**(소프트웨어)에 해당한다. 코드·아키텍처·기능을 공개하면 AI로 복제 가능.

**너의 비즈니스(서비스업)는 다르다:**

- 공개하는 건 "니치 분석 + 문제 정의 + 사이트 구조"
- 이걸 안다고 해서 "30곳 리서치 + 영업 + 제작 + 운영"을 똑같이 할 수 있는 사람이 있는가?
- **아니다.** moat(해자)은 정보가 아니라 실행력이다.
- 워크앤코 같은 에이전시가 케이스 스터디를 전부 공개하는 이유 = 공개 자체가 가장 강력한 영업 도구. 경쟁사가 봐도 똑같이 못 만듦.

### "고객 0인 상태에서 공개가 의미 있는가?" — 많은 사람이 묻는 질문

Product Hunt 토론에서 한 파운더가 핵심을 짚었다: "프로덕트 마켓 핏을 이미 알고 있으면 스텔스가 낫고, 아직 모르면 공개가 낫다."

너는 아직 고객 0. 프로덕트 마켓 핏 미확인 상태.\
→ **공개가 피드백을 빨리 받는 유일한 방법.**

### "나는 어떤 사람인가?" — 성향 체크

Mercury의 가이드가 잘 짚는다: "공개적 어카운터빌리티가 에너지를 주는 사람이 있고, 지치게 하는 사람이 있다."

→ 너는 사나에게 "자기 압박 장치"로 빌드인퍼블릭을 선택했다고 말했다. 이건 **공개가 맞는 성향**이라는 신호.

## 4. 추천: 하이브리드 — "과정은 Day 0부터, 포트폴리오는 첫 런칭 후"

두 전략의 장점만 조합하는 방법.

### 즉시 공개하는 것 (Day 0부터)

- **여정:** "오전 4시간, 월 500만 원 도전. 부티크 호텔 전용 웹사이트."
- **숫자:** 리드 수, 컨택 수, 회신율
- **칸반:** 보드 구조, 병목 분석
- **배운 것:** 리서치에서 발견한 부티크 호텔의 공통 문제 (구체적 호텔명 빼고)
- **도구·워크플로우:** 전부

### 첫 런칭 후 공개하는 것

- **포트폴리오/케이스 스터디:** Before/After 스크린샷 + 결과 수치 (고객 동의 후)
- **정확한 가격:** 첫 2\~3건의 가격 테스트 결과를 본 후에 공식 가격 공개
- **고객 이름:** 반드시 동의받은 후에만

### 절대 공개하지 않는 것

- 호텔 이름·담당자·연락처 (동의 전)
- 대화 내용, 계약서 상세 조건
- 고객사 내부 자료

## 5. "템플릿 복붙" 프레이밍 주의

공개할 때 가장 조심할 것 하나.

- ❌ "같은 템플릿 복붙" → Wix/Squarespace 템플릿 쓴다고 오해
- ✅ "부티크 호텔 30곳+ 리서치 기반으로 직접 설계한 전용 사이트 구조를 각 호텔에 맞춤 적용"

같은 사실이지만 프레이밍이 다르다. 거짓말이 아니라 정확한 표현. 이 차이가 "150만 원짜리"와 "500만 원짜리"를 가른다.

## 6. 결론 — 선례들이 공통으로 하는 말

**"숨기는 게 안전하다고 느끼지만, 실제로는 공개하는 쪽이 더 빨리 성장한다."**

Arvid: "실패와 실험이 잘못되는 것을 인정하는 게 진짜 의미 있는 신뢰 기반 장기 관계를 만든다."

Levels: "사람들은 불완전함과 실수를 공유하면 함께 여정을 가고 싶어한다."

Failory: "예전에는 아이디어를 도둑맞을까 봐 숨겼지만, 시간이 공유와 피드백의 이득이 복제 리스크를 압도한다는 걸 증명했다."

Learning Loop: "이상적으로는 스텔스 기간 0일. 회사를 시작하기로 한 순간부터 사람들에게 말해라."

**너의 상황 요약:**

- 서비스업이라 코드 복제 리스크 없음
- 혼자 하는 사업이라 자기 압박이 필수
- 고객 0이라 피드백 루프가 급함
- "오전 4시간, 월 500" 스토리 자체가 강력한 후크

→ **Day 0부터 여정을 열고, 첫 런칭 때 포트폴리오를 추가하는 하이브리드가 최선.**
