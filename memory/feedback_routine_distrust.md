사용자 발화 (2026-05-18, 정확 인용): **"나는 루틴이 제역할을 한다고 느낀적이 한 번도 없어"**.

STL 루틴 시스템(Closer/Ray Dalio/Sentinel/Engineer/Architect — 매일·매주 cron 자동 실행)을 사용자는 *작동한다고 체감한 적이 없다*. 같은 맥락에서 engineer 루틴은 폐지 결정 (2026-05-18).

**Why:** 루틴들은 실제로 자주 silent-fail 해왔다 — engineer exit 1, watcherMemory exit 127, ray-dalio 는 LaunchAgent 미설치, QUEUE P0 가 "closer/ray-dalio/engineer 매일 새벽 silent fail" 을 박아둠. 사용자가 한 번도 가치를 못 느낀 것은 정확한 관찰이다. 자동화는 "있다는 기분" 만 주고 실제 효용은 0에 가까웠다 (= 권유 장치의 또 다른 형태).

**How to apply:**
- 루틴 시스템을 *당연히 유지해야 할 자산* 으로 전제하지 말 것. 루틴을 건드리는 작업에서는 "이 루틴이 실제로 효용이 있나"를 먼저 의심한다.
- 새 자동화·routine 을 늘리는 제안에 신중. 효용 없는 자동화는 추가가 아니라 *삭제* 대상 (memory/feedback_single_solution.md — 효과 없는 권유 장치는 삭제).
- 루틴 폐지/축소 제안 시 사용자는 우호적 — MISSION.md 의 STL 5-루틴 구조는 고정 전제가 아니다.
- 단, 한 번에 전부 폐지하지 말고 사용자 확인을 받아 하나씩 (engineer 처럼).
