사용자 기준, 구현해본 적 없는 무언가를 처음 시도할 때, 아래 순서를 따른다.

**Why:** 바퀴를 재발명하지 말 것. 이미 동작 중인 케이스가 가장 신뢰할 수 있는 레퍼런스다.

**How to apply:** 막혔을 때가 아니라 **첫 시도부터 기본값으로** 이 절차를 수행한다. 구현보다 탐색이 먼저다. 시행착오를 기록한 뒤 한계와 구현 가치를 먼저 평가한다.

**레퍼런스 클래스의 범위 (중요):**
공식 docs/예제만이 아니라 아래까지 포함해야 진짜 레퍼런스 클래스 탐색임:
- 커뮤니티에서 "나 이렇게 해결했다" 류의 공유 포스트
- 댓글에서 "나도 이렇게 쓴다", "이 방법 좋더라" 반응이 달린 것
- best practice 또는 hack으로 커뮤니티에 형성된 패턴
- 일부 헤비유저에게라도 검증된 방법
"공식 예제에 없다"는 것만으로 판단 종료하지 말 것.

---

## 절차

1. **레퍼런스 클래스 집결지 탐색**: "이 케이스가 가장 많이 몰려있는 곳이 어딜까?" 하고 찾는다. Reddit, 해당 서비스 공식 커뮤니티, Discord, GitHub Issues, Stack Overflow 등 사용자가 많고 논의가 활발한 곳 기준으로 리스트업.

2. **적합도 순위에 따라 서칭**: 가장 관련성 높은 곳부터 순서대로 탐색.

3. **이미 정상 동작 중인 케이스 우선**: 실제로 사용 중인 구현을 찾는다. 케이스 수가 많을수록, 인정을 많이 받을수록 좋다.

4. **소스코드 직접 획득**: 구체적인 구현 방법과 소스코드가 있는지 확인. 있으면 그대로 가져와서 테스트. 이후 불편함이 생기면 조금씩 수정하며 사용.

5. **소스코드 없을 때**: 구두 설명 내용을 파악하고 추가 검색 → 그 방향으로 해결책 탐색.

6. **직접 결과 없을 때**: 핵심 로직 기준으로 비슷한 원리를 사용하는 다른 분야의 케이스를 생각하고 리스트화 → 탐색.

7. **유사 케이스 발견 시**: 3번으로 돌아가서 재시도.

8. **시행착오 기록 및 결과 정리**: 과정을 기록하며 수행하고, 결과를 먼저 요약 정리.

9. **한계와 구현 가치 평가**: 인사이트를 바탕으로 "한계가 있는지, 구현할 가치가 있는지" 먼저 묻는다.

10. **합의 후 구현 시작**: 주체적으로 만들어볼 가치가 있다고 합의된 후에만 구현을 시작한다.
