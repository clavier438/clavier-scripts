문제를 풀 때 여러 장치·검사·알람을 덧대 "막았다는 기분"을 만들지 않는다. **나쁜 상태로 가는 경로 자체가 존재하지 않게 하는, 다른 path 없는 단순한 해결책 하나**를 찾아 그것만 적용한다.

**Why:** 덧댄 층은 각각 샌다 — 합쳐도 "감시"일 뿐 "불가능"이 아니다. 여러 장치는 실제 안전이 아니라 *안전하다는 느낌*(false confidence)만 강화한다. 의지·규율에 기댄 해결책은 나쁜 상태가 *도달 가능*한 한 언젠가 무너진다. (2026-05-18 SSOT drift 사고: gitSync 데몬이 규율을 자동화하려다 거짓 "push 완료"까지 보고 → Claude의 4계층 감시 안을 사용자가 기각.)

**How to apply:**
- 문제를 만나면 "어떤 장치를 더할까"가 아니라 "나쁜 상태로 가는 *경로를 어떻게 삭제할까*"를 먼저 묻는다.
- 해결책이 여러 개로 쪼개지면 아직 진짜를 못 찾은 것 — 계속 단순화한다.
- 100%가 물리적으로 불가능하면 "100%에 가까운" 하나를 택하되, 못 막는 잔여는 *침묵*이 아니라 *시끄러운 실패*로 드러나게 한다.
- 이 원칙은 메모리·아키텍처·워크플로 등 **모든 설계 판단에 우선**한다.
