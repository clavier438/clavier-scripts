문제를 풀 때 여러 장치·검사·알람을 덧대 "막았다는 기분"을 만들지 않는다. **나쁜 상태로 가는 경로 자체가 존재하지 않게 하는, 다른 path 없는 단순한 해결책 하나**를 찾아 그것만 적용한다.

**Why:** 덧댄 층은 각각 샌다 — 합쳐도 "감시"일 뿐 "불가능"이 아니다. 여러 장치는 실제 안전이 아니라 *안전하다는 느낌*(false confidence)만 강화한다. 의지·규율에 기댄 해결책은 나쁜 상태가 *도달 가능*한 한 언젠가 무너진다. (2026-05-18 SSOT drift 사고: gitSync 데몬이 규율을 자동화하려다 거짓 "push 완료"까지 보고 → Claude의 4계층 감시 안을 사용자가 기각.)

**How to apply:**
- 문제를 만나면 "어떤 장치를 더할까"가 아니라 "나쁜 상태로 가는 *경로를 어떻게 삭제할까*"를 먼저 묻는다.
- 해결책이 여러 개로 쪼개지면 아직 진짜를 못 찾은 것 — 계속 단순화한다.
- 100%가 물리적으로 불가능하면 "100%에 가까운" 하나를 택하되, 못 막는 잔여는 *침묵*이 아니라 *시끄러운 실패*로 드러나게 한다.
- 이 원칙은 메모리·아키텍처·워크플로 등 **모든 설계 판단에 우선**한다.

**효과 본 장치 = 훅. "~하자"/지침/감사/알람 = 권유 = 무효.** 사용자가 기억하는 유일하게 효과 있던 장치는 훅이다 (commit⇒push 가 "로컬≠원격"을 *불가능*하게 만든 것처럼). 새 장치를 고안할 땐 "이게 의지에 기대나?"를 먼저 묻는다 — 기대면 권유다. 효과 없는 권유 장치는 **그냥 삭제**해도 좋다 (사용자 명시 허락 2026-05-18).

**현재상태 정보를 손으로 쓰는 것 자체가 권유다.** 손으로 쓴 "현황 도면/표/문서"는 현실의 *사본*이다. 사본은 둘이면 반드시 어긋난다 — "바뀌면 같이 갱신" 규칙은 의지에 기댄 권유. 구조적 해법 = 현재상태는 *쓰지 않고 생성/계산*한다 (실제 소스에서 렌더링, SQL VIEW 가 테이블의 사본이 아니듯). 생성된 것은 사본이 아니라 현실 자체라 drift 불가. 손으로 남길 것 = intent/decision/concept (왜·방향) 뿐 — 이건 "현재 기계 상태" 주장이 아니라 drift 개념이 없다.

**조금 불완전해도 *지속될 수밖에 없는* 구조를 택한다.** 완벽하지만 의지에 기댄 구조 < 덜 예쁘지만 어긋날 수 없는 구조. (생성된 도면이 손으로 큐레이션한 것보다 못생겨도, 어긋날 수 없으므로 그쪽을 택한다.)
