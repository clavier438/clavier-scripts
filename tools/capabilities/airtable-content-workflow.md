## 콘텐츠 워크플로우 8단계 (★)

사용자 발화 트리거:
- "○○ 호텔 콘텐츠 만들어줘"
- "○○ 페이지 콘텐츠 추가"
- "이 base 에 콘텐츠 박아줘"
- "stay/숙소 사이트 콘텐츠"

```
1. 컨텍스트 수집
   - 브랜드/장소 정보 (web search, hubspot, 인스타 등)
   - 의사결정자 접점 (필요 시)

2. 레퍼런스 base 구조 분석
   - 모델 base 의 CSV 구조 fetch (테이블/필드/관계)
   - **분류 인스턴스** (group/tags/topics records) 의 의미 파악
   - 호텔 도메인 보편 패턴 — 기존 분류 그대로 재사용 가능한지 확인

3. 컨셉 도출 (한 줄)
   - 본질 = "무엇을 파는가" (예: "로컬의 로컬을 판다", "한 동네 전체를 경험")
   - 사용자와 합의

4. 카피 작성 — CSV 이전, 텍스트만
   - 페이지별 (객실/식사/로비/체험/이웃들 등)
   - 카테고리 문장 = 공통 성질 한 번만
   - 아이템 = 차별적 디테일 + 최소 스펙
   - 메뉴판 패턴 = 헤딩 + 리스트 (구체 예시로 상상 유발)
   - **마크다운 본문으로** 사용자에게 보여줌 (CSV X)

5. 검수 / 디렉션 iterate
   - 사용자가 톤/구체도/구조 짚어 줌 → 다시
   - 합의될 때까지 반복 (CSV 이전 단계)

6. CSV 변환 — 합의 후
   - 레퍼런스 base 구조 *그대로* (테이블/필드/타입)
   - 데이터만 새 콘텐츠로
   - 새 분류 인스턴스 = 사용자 합의된 *진짜 필요* 한 것만
   - 각 record `slugKey` 영문 stable key 부여
   - link 컬럼 = target 테이블의 slugKey 들, `|` 구분
   - formula/lookup/autoNumber/rollup 컬럼은 빼고

7. data_dir 저장 + 사용자 검수
   - `<base 폴더>/sisoso_v6/` 같은 곳에 `<table>.csv` 파일별
   - dry-run 으로 어떤 변경 일어날지 사용자 확인

8. V6 push (실제 주입)
   - `AIRTABLE_PAT=... node airtableUpsertV6.mjs <base> <dir>`
   - 검증 (record count, link 확인)
```

**❌ 사나(외부 LLM) 거치지 마**. 사나 무한 토큰 워크플로우는 폐기 (잘림 + 4-step 마찰).
**Claude 가 자료조사부터 push 까지 한 흐름** — 사용자 의사결정만 받음.

## 카피 작성 원칙 (★ 4단계 카피 작성 시 절대 원칙)

사용자가 사나에게 반복해서 짚은 원칙들. 미래 콘텐츠 작업에도 그대로 — 다만 *프로젝트마다 진화 가능*.

### 원칙 1: 컨셉 설명 X, 컨셉대로 말함
- "우리는 로컬을 판다" 같은 *메타-말* 안 쓴다
- 컨셉을 *말하지* 말고 컨셉*대로* 쓴다 → 독자가 자연스럽게 느낀다
- 예시:
  - ❌ "이곳은 시설보다 풍경을 강조하는 객실입니다" (메타 설명)
  - ✅ "이 방은 파도가 바로 아래에서 부서지는 소리가 잘 들리는 층에 있습니다" (그냥 말함)

### 원칙 2: 공통 / 차별 분리
- 카테고리 문장 = **모든 아이템 공통 성질** 한 번만
- 아이템 문장 = **그 공통을 다른 각도로** 푼다 (반복 X)
- 예시 (객실):
  - 카테고리: "어디에 머물든 창을 통해 삼척 앞바다의 파도와 수평선을 한눈에 보게 됩니다."
  - 아이템 1: "파도 소리가 가장 가까이 들리는 층" (낮은 층 차별)
  - 아이템 2: "수평선이 곧게 이어져 보이는 높은 층" (높은 층 차별)

### 원칙 3: 본질 vs 수단
- 시설/소개방식/공간구조 = **수단**. 본질이 아님
- 본질 = 그 자리가 *진짜 파는 가치* (사람, 물건, 시간, 경험)
- 예시 (로비):
  - ❌ "로비는 단순한 휴게 공간이 아니라, 로컬 파트너의 물건을 경험할 수 있는 마켓입니다" (시설 설명)
  - ✅ "어부가 햇볕 아래에서 뒤집어 가며 말린 김, 산을 오르내리며 모은 잎과 열매로 블렌딩한 차" (본질 = 사람과 물건)

### 원칙 4: 메뉴판 패턴 — 구체 예시로 상상 유발
- 일반 정보 ("계절 식재료") X
- 구체 예시 ("봄 해초 모둠: 파래·미역·톳을 살짝 데쳐 유자 간장으로만 무친 전채") O
- 속담의 돌과 새처럼 — 구체가 추상보다 강력함
- 헤딩 + 리스트 형식 (메뉴판처럼)

### 원칙 5: CSV 이전 카피 검수
- 카피 *합의 전에 CSV 구조화 금지*
- 마크다운 본문 → 사용자 디렉션 → iterate → 합의 후 CSV
- CSV 단계에서 카피 톤 바꾸려면 두 곳 (마크다운 + CSV) 동기화 짐

### 원칙 6: dynamic binding 최소화
- `{variable}` 같은 템플릿 변수 X (Framer 바인딩 어려움)
- CTA = 보편적 텍스트 ("자세히 보기", "예약 문의", "지도에서 위치 보기")
- 카테고리 공통으로 써도 어색 X

## 분류 인스턴스 재사용 규칙

호텔 도메인 패턴이 보편적이라 — *기본은 기존 분류 활용*:

- ✅ default: 모델 base 의 group/tags/topics 그대로 활용 — 데이터만 갈아끼움
- ⚠️ 예외: 진짜 필요할 때만 새 분류 신설 (예: mukayu 에 없는 `neighborhood` 같은 컨셉)
- ❌ 절대 X: 같은 의미인데 새 이름으로 분류 만들기 (예: `food-list` 신설했는데 사실 `cuisine-list` 와 같은 의미)

판단:
- mukayu 의 분류 인스턴스 = mukayu *콘텐츠의 인스턴스* (단순 빈 슬롯 아님)
- sisoso 콘텐츠도 같은 호텔 도메인 — *대부분 매핑 가능*
- 매핑 안 되는 컨셉만 신설
